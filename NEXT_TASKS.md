# NEXT_TASKS — 도메인 확정 전 진행 가능 작업

> 기준 문서: `PROJECT.md` · 작성 기준: 현재 저장소 실제 구현 상태
>
> **이 목록의 원칙**: 도메인이 바뀌어도 다시 할 필요가 없는 작업만 담았습니다.
> 도메인은 `sites/andrology/hospital.json`의 `site.url` 한 곳에서만 관리되므로,
> 아래 작업들은 나중에 도메인이 확정되면 `site.url` 값 하나만 바꾸면
> canonical·OG·sitemap·robots가 전부 자동 갱신됩니다.

---

## 우선순위 1 — 지금 바로 (콘텐츠 자산 축적)

### 1-1. 타깃 키워드 전략 수립

- **목적**: 어떤 검색어로 유입시킬지 먼저 정해야 아티클 발행이 흩어지지 않음. 도메인과 무관하게 지금 확정 가능
- **하는 일**: 핵심 진료 분야 기준 타깃 키워드 20~30개 목록화, 키워드별 slug 미리 지정, 발행 순서 결정
- **완료 조건**: 키워드·slug·발행 순서가 정리된 목록 문서 확보 (예: `docs/keyword-plan.md` 또는 별도 메모)

### 1-2. 아티클 발행 루프 운영 시작

- **목적**: 검색 노출은 콘텐츠 양과 시간이 좌우 — 도메인 확정을 기다리며 콘텐츠를 미리 쌓아두는 것이 가장 남는 작업
- **하는 일**: `npm run draft-article` → Claude에 프롬프트 입력 → `npm run import-ai` 반복 (이미 검증된 루프)
- **완료 조건**: 아티클 5편 이상 등록 + `npm run build` 로컬 통과 + 의료 내용 검토 완료 (push·배포는 도메인 확정 후 일괄 진행 가능)

### 1-3. 사이트 기본 콘텐츠 실제 값 채우기

- **목적**: 현재 소개문·진료과목·FAQ·Hero 문구가 전부 "예시" 상태 — 실제 문구로 교체해야 SEO 경고 해소 및 영업 데모 품질 확보
- **하는 일**: hospital.json의 description(50~160자), services 실제 진료과목, faq 실전 질답, hero·cta 문구 작성 (Claude에게 초안 요청 가능)
- **완료 조건**: `npm run check:seo`에서 "예시" 관련 콘텐츠 경고 해소, 화면에 예시 문구 0건

## 우선순위 2 — 이번 주 내 (신뢰도·브랜딩)

### 2-1. 브랜딩 실제 값 적용

- **목적**: 대표 색상·로고가 "우리 병원처럼 보이는" 체감의 핵심 — 이미 기능은 완성되어 있고 값만 넣으면 됨
- **하는 일**: 로고 파일을 `public\images\`에 넣고 `images.logo` 경로 지정, `theme.primary`에 병원 상징색 지정
- **완료 조건**: 로컬(`npm run dev`)에서 이미지 로고·색상 반영 확인, check:seo의 로고 경고 해소

### 2-2. 병원 기본 정보 확정 반영 (주소 제외)

- **목적**: 전화번호·운영시간·대표원장이 "미정"이면 상담 버튼과 구조화 데이터가 비어 있음 — 확정 가능한 것부터 반영
- **하는 일**: phone·hours·doctor·channels(카카오 채널 등) 값 입력. ※ 주소·지도는 확정 후 진행
- **완료 조건**: check:seo의 "미정" 경고 감소, 전화 상담 버튼 표시, JSON-LD에 telephone 포함

### 2-3. 의료진 소개 기능 개발 (doctors 배열)

- **목적**: 의료 콘텐츠의 신뢰도(E-E-A-T)와 병원 영업 시 필수 요구 항목 — 도메인과 무관한 코드 작업
- **하는 일**: hospital.json에 `doctors` 배열(이름·직함·사진·경력) 추가 + 렌더링. 기존 `doctor` 문자열과 호환 유지
- **완료 조건**: JSON만으로 의료진 여러 명 표시, 기존 사이트 빌드 결과 무변화(하위 호환), SEO 검사 통과

## 우선순위 3 — 여유 시 (플랫폼 확장 준비)

### 3-1. 홈 섹션 순서·표시 제어 (homeSections)

- **목적**: 업체별 홈 구성 차별화 — 멀티 사이트 영업의 커스터마이즈 폭 확대
- **하는 일**: `"homeSections": ["hero","about","services","faq","contact"]` 배열로 순서·숨김 제어 구현 (없으면 현재 순서 유지)
- **완료 조건**: 배열 변경만으로 홈 구성이 바뀌고, 항목이 없을 때 기존과 동일 렌더링

### 3-2. 두 번째 업체 사이트 로컬 실증

- **목적**: 멀티 사이트 엔진의 실전 검증을 배포 없이 미리 완료 — 도메인 확정 즉시 배포만 하면 되는 상태 만들기
- **하는 일**: `npm run create-site`로 테스트 업체 생성 → `SITE=업체ID npm run build` 로컬 빌드 검증 (Cloudflare 연결은 하지 않음)
- **완료 조건**: 두 번째 사이트가 로컬에서 독립 빌드 성공, 절차 소요시간 기록

### 3-3. 아티클 데이터 분리 설계 메모

- **목적**: 아티클이 10편을 넘으면 hospital.json이 비대해짐 — 미리 분리 방식을 정해두면 나중에 무중단 전환 가능
- **하는 일**: 분리 구조(예: `sites/업체/articles/*.json`)와 전환 절차를 짧은 설계 메모로 작성 (구현은 나중)
- **완료 조건**: 분리 기준·구조·전환 순서가 담긴 메모 1페이지

## 도메인 확정 후로 미루는 작업 (이번 목록에서 제외)

- Cloudflare Pages 프로젝트 생성·도메인 연결 (신규 사이트)
- `site.url` 변경 및 canonical·sitemap·robots 재확인
- 구글 서치콘솔·네이버 서치어드바이저 등록 및 사이트맵 제출
- 지도·오시는길(location) 기능 — 주소 확정 필요
- 방문자 분석 도구 연결

## 진행 순서 제안

1-1 키워드 전략(반나절) → 1-3 기본 콘텐츠(1일) → 1-2 아티클 발행(상시 반복) 순으로 시작하고,
2번 그룹은 병원 정보·로고가 준비되는 대로, 3번 그룹은 개발 여력이 있을 때 진행하시면 됩니다.
핵심은 **1-2를 멈추지 않는 것** — 도메인 확정 시점에 이미 콘텐츠가 쌓여 있어야
배포 즉시 색인·노출 게임을 시작할 수 있습니다.
